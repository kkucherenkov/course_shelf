## Notes

- Completed: 2026-04-27 (retroactive flip via T-2026-04-27-030)
- **Deviation: the `@app/api-client-ts/server` subpath with class-validator-decorated DTO classes was deliberately not implemented.** Instead, runtime validation is handled by `express-openapi-validator` middleware (E04-F02-S03) which validates every request body, query, and response against the bundled `dist/openapi.json` at the HTTP boundary. Backend handlers consume the contract via type-only imports from `@app/api-client-ts` (e.g. `import type { RecordProgressRequest } from '@app/api-client-ts'`).
- Why: same runtime guarantee from a single source of truth (the spec) instead of two (the spec **and** decorators on generated classes). Eliminates the drift risk of decorators not matching schema constraints. The validator middleware also produces RFC 9457 `application/problem+json` errors for free.
- Trade-off: backend handlers can't `validateSync(dto)` ad-hoc — they trust the validator middleware. In practice every entry point is a Nest `@Controller` route, so the middleware runs first. Direct `CommandBus.execute(new SomeCommand(...))` calls inside other handlers don't get re-validated, but they're already type-checked against the same generated types.
