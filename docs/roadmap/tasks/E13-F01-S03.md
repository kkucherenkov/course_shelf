## Notes

- Completed: 2026-04-27
- Result: T-2026-04-27-037 — four feature commits on `feat/form-primitives`: `e28d7ec` (audit/fix), `c8d232d` (AppCheckbox), `3aee420` (AppRadio + AppRadioGroup), `9136d42` (composites + barrel). UI tests 352/352 (+80 new cases); web tests 17/17; lint + typecheck clean.
- Audit findings: `AppInput` heights used `var(--space-20)` etc. — non-existent tokens, so heights were unset; `AppField`'s error color used the non-existent `--status-danger`; `AppSwitch`'s labeled branch had `inheritAttrs: true` so AppField's `id`/`aria-*` slot attrs landed on the `<label>` instead of the `<button role='switch'>`. All three fixed.
- `AppRadio` arrow-key nav uses direct DOM traversal (`querySelectorAll('[role="radio"]')`) keyed off the parent `[role="radiogroup"]` — keeps `AppRadioGroup`'s API surface minimal and matches the WCAG radiogroup pattern. Only the checked radio carries `tabindex="0"` so Tab moves into the group exactly once and arrow keys cycle within.
- `AppCheckbox` supports `indeterminate` via `aria-checked="mixed"`. From indeterminate the next toggle goes to `true` (matches native HTML behaviour).
- `AppNumberField` uses `AppIconButton` (size sm, ghost variant) for the `+`/`−` steppers; clamps to `min`/`max`; stepper buttons disable at boundaries.
- `AppSearchField` shows leading `IconCS name="search"` (decorative) and a trailing `AppIconButton name="x"` clear button (visible only with a value, `aria-label` overridable).
- ESLint config tweak: `vue/attribute-hyphenation` ignores `ariaLabel` so AppIconButton's prop can be written camelCase in templates. Vue maps `aria-label` ↔ `ariaLabel` at runtime, but vue-tsc requires the camelCase form in static types. Scoped to `packages/ui`.
- Token deviations carried over from earlier work: `--surface-surface-alt` → `--surface-raised`, `--status-danger` → `--status-error-fg`, `--text-fg-inverse` → `--text-inverse`. Documented inline.
