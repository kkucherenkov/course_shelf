# Internationalisation

No user-visible string may ship as a literal. Ever. Three platforms, three implementations — same discipline.

Default locale: `en`. Every key in `en` **must** exist in `ru` (and `el`, `uk` where added). Missing translation is a bug, not a fallback.

## Web — `@nuxtjs/i18n` + global TS locale modules

Layout (everything under `<rootDir>/i18n/`, the v9+ convention):

```
apps/web/i18n/
├── i18n.config.ts              # defineI18nConfig — vue-i18n options + messages
└── locales/
    ├── en.ts                   # export default { ... } — message tree
    └── ru.ts
```

`apps/web/i18n/i18n.config.ts`:

```ts
import en from './locales/en';
import ru from './locales/ru';

export default defineI18nConfig(() => ({
  legacy: false,
  locale: 'en',
  fallbackLocale: 'en',
  messages: { en, ru },
}));
```

`apps/web/nuxt.config.ts`:

```ts
i18n: {
  vueI18n: './i18n.config.ts',  // resolved relative to <rootDir>/i18n/
  strategy: 'no_prefix',
  defaultLocale: 'en',
  locales: [
    { code: 'en', language: 'en-US', name: 'English' },
    { code: 'ru', language: 'ru-RU', name: 'Русский' },
  ],
},
```

Use the global scope:

```ts
const { t } = useI18n();
// ...
t('pages.login.title');
t('layouts.default.appName');
```

Keys are **namespaced by where they're used** — `pages.<page>.<key>`,
`layouts.<layout>.<key>`, `components.<component>.<key>`. New screens
extend the message tree in `en.ts` + `ru.ts`.

- `@app/ui` components **never** call `useI18n()` — receive translated strings via props.
- To embed a literal `@`, escape: `{'@'}` (vue-i18n linked-message syntax).
- The `pnpm check:i18n` script verifies key parity between `en.ts` and `ru.ts` (every key in `en` exists in `ru`).

### Why TS modules and not JSON files (`langDir` + `file:`)

Three failure modes pushed us off the more common `langDir`-based config:

1. **SFC `<i18n>` blocks**: `vite:json` claims the `?vue&type=i18n&...lang.json` URL
   before `@intlify/unplugin-vue-i18n` can — so the SFC pattern that worked on
   v8/v9 + webpack throws "Failed to parse JSON file" on every page mount under
   v10 + Vite.
2. **Per-locale JSON files via `langDir` + `file:`**: the unplugin's `include`
   gets `localePaths` (absolute path strings used as picomatch patterns) which
   don't reliably match the absolute path Vite passes during transform — so
   `vite:json` wins again.
3. **JSON imports inside `i18n.config.ts`**: even when reached via the standard
   module graph, the unplugin claims them and pre-compiles to a runtime that
   doesn't expose `default` as a plain message object — `messages.{en,ru}` reach
   vue-i18n empty and every `t()` returns the key string.

Plain `.ts` modules with `export default { ... }` sail through Vite's standard
TS transform untouched by either plugin. The default export is the real
messages object.

### Common gotchas

- `i18n.config.ts` must live at `apps/web/i18n/i18n.config.ts`. The `vueI18n`
  path in `nuxt.config.ts` is resolved relative to `<rootDir>/i18n/`, not
  `<rootDir>/`. Putting it at the project root silently fails — Nuxt loads no
  messages and `t('key')` returns the key string.
- Locale files **must be `.ts`**, not `.json` (see "Why TS modules" above).
  `as const` after the object literal preserves narrow types if you ever want
  type-safe keys.
- Don't mix `langDir`+`file:` with `vueI18n` config — pick one. We use
  `vueI18n` exclusively.
- Don't add `<i18n>` SFC blocks anywhere. They'll re-trigger the v10 + Vite
  parse race.

## Mobile — `slang` with namespace files

Files: `lib/i18n/<namespace>_<locale>.i18n.json` — one file per `(namespace, locale)` pair.

Config (`slang.yaml`): `namespaces: true`, `string_interpolation: braces`, `enum_name: AppLocale`, `flutter_integration: true`.

Regenerate: `dart run slang` (or via `build_runner`). Output `strings.g.dart` + `strings_<locale>.g.dart` — never edit by hand, gitignored.

```dart
// main()
LocaleSettings.useDeviceLocale();
runApp(TranslationProvider(child: const ProviderScope(child: MyApp())));
```

`MaterialApp` gets `locale: TranslationProvider.of(context).flutterLocale` + `supportedLocales: AppLocaleUtils.supportedLocales`.

Consume: `context.t.<namespace>.<key>` — e.g. `context.t.common.appTitle`. No raw `Text('Overall')` — review blocker.

## Backend — `nestjs-i18n` with per-locale namespace files

Files: `apps/backend/src/i18n/<locale>/<namespace>.json`. One subdirectory per language, one file per module. Copied to `dist/i18n/` via `nest-cli.json` assets rule.

`I18nModule.forRoot` in `app.module.ts`: `I18nJsonLoader`, `fallbackLanguage: 'en'`, resolvers: `QueryResolver(['lang'])`, `HeaderResolver(['x-lang'])`, `AcceptLanguageResolver`.

```ts
const lang = I18nContext.current()?.lang;
throw new UnauthorizedException(
  this.i18n.t('realtime.signInRequired', lang ? { lang } : undefined),
);
```

Translation key: `<namespace>.<key>` (filename = namespace prefix).

**Error bodies** use RFC 9457 `application/problem+json` — `title` and `detail` stay English-canonical. `nestjs-i18n` is for server-generated strings clients pass through verbatim.
